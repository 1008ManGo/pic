# **企业级短信平台完整功能清单**

## **一、管理员模块（系统控制与配置）**
### 1️⃣ 上游通道管理 (下行网关)
* 新增/编辑/删除 SMPP 上游通道
* 配置通道参数：
  * 账号、密码、Host、Port
  * 最大TPS、最大绑定数
  * 绑定/系统类型(可空)
  * 心跳时间/断线重连等等
* 通道监控：
  * 在线/离线状态
  * 当前队列长度
  * 消息成功率/失败率
  * TPS 实时监控

### 2️⃣ 用户管理
* 添加/编辑/禁用用户
* API Key / Token 管理
* 用户权限管理：
  * 分配可用通道
  * 允许发送国家列表
  * 固定分配发件人ID或开启用户自定义发件人ID
  * 每个国家独立定价
  * 账户余额

### 3️⃣ 计费与套餐管理员模块
* 用户单独国家定价设置
* 消息失败扣费策略（是否返还扣费）
* 充值与扣费记录
* 查看财务报表（按用户、国家、时间段）
* 管理发件人ID

### 4️⃣ 系统设置
* 站点名称等基本设置
* 异常告警策略配置

---

## **二、用户模块（短信发送与统计）**
### 1️⃣ 短信发送功能
* 用户可提交短信：
  * 单条、上传txt文件批量发送里面的号码，单文件最大100万个号码，文件内号码都必须带国际码
  * 参数：发件人ID、目标号码、内容、优先级
  * 内容框实时显示当前编码/字符/计费条数(如果开启附加随机字符也要计入字符)
  * 可选附加随机字符(避免相同内容大规模屏蔽,开启时短信内容最后面附加随机5位大小写字母+数字)
  * 发送前系统自动校验：
  * 用户是否有该国家发送权限
  * 号码合法性（国际码 + 本地长度)
①短信内容只包含GSM7编码字符时，内容160字/条，超出160字符按153字/条，其中GSM7 标准字符，每个字符按1个计算字符数；GSM7 扩展字符，每个字符按2个计算字符数

②短信内容中包含除GSM7编码字符时，内容按70字/条，超出70字按67字/条拆分；

* 系统自动：
  * 拆分长短信（GSM7 / UCS2）
  * 按用户国家定价扣费
  * 入队异步发送到分配的 SMPP 通道

### 2️⃣ 消息查询与管理
* 查询单条/批量消息状态（成功/失败/未知/排队中/发送中等等状态）
* 查看历史发送记录
* 导出发送报表（Excel/csv）
* 消息补发（失败或未发送）

## **七、消息完整流程**

1. 用户提交短信（选择目标国家）
2. 系统校验：
   * 用户权限（国家）
   * 号码合法性
   * 余额是否足够
3. 系统拆分长短信 → 按用户国家定价计费
4. 入队异步发送 → 分配到用户可用的 SMPP 通道
5. 接收 Delivery Report → 更新状态
7. 用户通过后台/API 查询状态或导出报表

---

✅ **系统特点总结**：
* 上游通道不分国家，灵活分配
* 用户层面独立国家权限与定价
* 全流程计费、拆分、状态
* 高并发、可监控、可告警、可审计
* 支持扩展分布式、多通道、多渠道
* 计费为提交计费
* 短信状态分为上游返回状态和内部状态
* 内部状态比如未发送，通道异常导致发送失败，发件人ID不合法，余额不足导致未发送等等

# **一、整体技术架构（推荐）**

👉 系统本质是：

```
API服务 + 业务逻辑 + SMPP客户端 + 队列 + 数据库 + 管理后台 + 监控
```

# **二、后端技术栈（核心推荐）**

## ✅ **首选：C# (.NET 8)**

### 1️⃣ Web API层

* **ASP.NET Core Web API**

  * 高性能（比Node/Python更稳）
  * 自带认证、中间件、日志
  * 非常适合你这种“计费 + 规则复杂”的系统

---

### 2️⃣ SMPP通信层

* 推荐：

  * **Inetlab.SMPP（强烈推荐）**
* 负责：

  * Bind 上游
  * SubmitSm发送
  * DeliverSm（上行 + 状态回执）

---

### 3️⃣ 队列系统（非常关键）

* **RabbitMQ **

  * 高可靠

---

### 4️⃣ 数据库

  * MySQL
---

### 5️⃣ ORM

* **Entity Framework Core**

  * 开发快
  * 适合复杂业务（计费、用户、国家定价）

---

### 6️⃣ 缓存

* **Redis**

  * 用户权限缓存
  * 国家价格缓存
  * TPS计数器
  * 防重复提交

---
